新浪军事任务创建试用记录
========================

本文记录一次以媒体运营者视角完成的真实任务创建试用。目标站点为 ``https://mil.news.sina.com.cn/``，验收标准是任务真实保存成功，且“测试索引”“测试新闻”都返回正确结果。


试用目标
--------

试用围绕下面这条主线展开：

- 先确认入口页能找到真实文章链接
- 再确认一篇文章能提取标题、正文、时间、来源
- 最后检查配置并保存任务

试用关注点不是系统内部细节，而是“一个不关心 XPath 和正则的内容运营者，能否顺着页面把任务建出来”。


最终保存成功的配置
------------------

本次实际保存成功的一组配置如下：

- 任务名称：``新浪军事``
- Spider：``browser``
- Parser：``IndexParser``
- 入口 URL：``https://mil.news.sina.com.cn/``
- 索引 XPath：``//div[contains(@class,'ty-cardlist-w')]//h3[@class='ty-card-tt']/a[@node-type='sima-click-0']/@href``
- 标题 XPath：``//h1[@class="main-title"]//text()``
- 正文 XPath：``//*[@id="article"]//p//text()``
- 时间 XPath：``//div[@class="date-source"]//span[@class="date"]//text()``
- 来源 XPath：``//div[@class="date-source"]//a[@class="source"]//text()``


试用结果
--------

- 任务已真实保存到任务列表
- “测试索引”成功返回多条真实新浪军事文章 URL
- “测试新闻”成功返回 ``url``、``title``、``content``、``src_time``、``source``
- 保存后重新进入编辑页，配置值与测试通过时一致

这说明当前项目已经可以完成复杂新闻站点的任务创建，但主路径中仍然存在多个会误导用户或增加成本的环节。


试用中确认的问题
----------------

阻塞型问题
~~~~~~~~~~

1. 入口页规则生成结果与 ``IndexParser`` 的真实语义不一致

   - 预览高亮可以直接使用选中的 ``<a>`` 节点 XPath
   - ``IndexParser`` 实际需要的是可直接提取 URL 字符串的 XPath，例如 ``/@href``
   - 用户看到高亮正确，但“测试索引”“测试新闻”会直接失败

2. 文章字段生成的成功提示过于乐观

   - 页面提示“标题、正文、时间、来源已写入配置”
   - 实际表单中可能仍有字段为空，尤其是 ``src_time``、``source`` 这类短文本字段
   - 用户会误以为已经完成，只在测试时才发现配置并未真正落表


高摩擦问题
~~~~~~~~~~

1. 任务名自动建议容易命中页面噪音

   - 入口页第一次探索时，任务名曾被自动填成 ``icon关闭``
   - 这类结果不符合“按站点 / 频道创建任务”的运营心智

2. 生成规则、生成字段和测试时的等待反馈不足

   - 页面只显示“正在生成……”或“正在执行测试……”
   - 用户无法区分当前是在读取页面、分析点选结果，还是在等待模型返回

3. 文章页时间与来源节点不容易稳定点选

   - 这类节点文本短、结构紧，容易点到外围容器
   - 一旦选中的是容器节点，模型生成结果就容易偏到错误类名或错误层级


体验一致性问题
~~~~~~~~~~~~~~

1. 编辑页重新打开后，顶部状态容易回到“等待探索”的表达

   - 用户已经保存过任务，但页面仍像刚进入时那样提示等待探索
   - 这会削弱“已经有一份可工作的配置”这一事实

2. 测试结果之前更像原始数据展示，不够直接回答“现在能不能用”

   - 用户更关心字段有没有出来、链接是不是对的、当前配置值是什么
   - 不关心原始 HTML 拼接结果或模型内部过程


优化方向
--------

围绕这次试用，任务创建主路径应继续收敛到下面几条原则：

- 预览规则和解析规则分离，页面高亮与真实抓取结果都要正确
- 字段生成后先校验真实落表结果，再给用户成功提示
- 测试结果直接展示“抓到什么、缺什么、当前配置是什么”
- 默认任务名优先体现站点或频道，不再直接依赖页面 title
- 页面打开时直接反映当前任务所处阶段，而不是一律从空白状态开始


当前判断
--------

这次新浪军事试用表明，当前项目最需要继续优化的不是“能不能抓”，而是“用户是否能稳定完成创建并确认结果”。当页面预览、规则回填、测试结果和保存动作真正统一后，任务创建页才能成为内容运营者可以放心使用的工作台。
