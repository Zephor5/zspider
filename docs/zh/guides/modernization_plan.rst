现代化改造规划
==============

本文给出 ZSpider 的一条务实现代化路径。

目标不是把项目推倒重写，而是在保留现有产品形态和领域模型的前提下，逐步降低维护成本、部署摩擦和安全风险。

为什么要改造
------------

从现有代码和分析文档来看，ZSpider 已经具备一套有价值的平台形态：

- 清晰的三进程运行结构
- 围绕任务的运营模型
- 配置驱动的解析能力
- 可用的后台管理界面

问题主要不在产品方向，而在工程基础设施：

- 依赖较旧且对版本敏感
- 环境配置写死在代码里
- 安全基线偏弱
- Web、后台和运行时服务边界不够清晰
- 可观测性和发布流程不够现代化

因此，改造优先级应当放在运行时、安全性和可维护性上。

指导原则
--------

1. 保留领域模型。

   ``Task``、``Parser``、``ArticleField``、``Dispatcher``、``Crawler`` 和结果查看能力是项目的核心价值，不应在改造过程中被模糊掉。

2. 增量替换优于重写。

   全量重写会带来较高迁移风险，也会拖慢交付。这个项目更适合分阶段升级，并保留兼容窗口。

3. 区分产品问题和技术债问题。

   现有产品定位仍然成立，改造的重点应该是工程债，而不是先重构业务概念。

4. 让部署和运行变得可重复。

   本地启动、测试执行、CI 和生产配置都应当可复现、可说明、可自动化。

5. 去掉高风险的灵活性。

   比如来自配置的任意 ``eval`` 执行，应替换为受约束、可审计的机制。

目标状态
--------

改造完成后，ZSpider 应当具备以下特征：

- 支持现代 Python 运行时
- 配置完全环境化
- 依赖可锁定、可复现
- 后端接口具备 API-first 边界
- 更稳健的认证与授权能力
- 更安全的发布和转换规则
- 具备结构化日志、指标和健康检查
- CI 能稳定执行测试和质量检查
- 运维迁移路径有文档支撑

应当保留的部分
--------------

+----------------------+----------------------------------------+
| 领域                 | 应当保留                               |
+======================+========================================+
| 运行形态             | ``web`` / ``dispatcher`` / ``crawler`` |
+----------------------+----------------------------------------+
| 产品模型             | 面向持续任务的抓取平台                 |
+----------------------+----------------------------------------+
| 解析抽象             | parser + field 配置模型                |
+----------------------+----------------------------------------+
| 异步执行             | 调度与 worker 解耦                     |
+----------------------+----------------------------------------+
| 自托管后台           | 内部运营和监控场景                     |
+----------------------+----------------------------------------+

优先改造项
----------

优先级应按照运营风险而不是“技术是否新潮”来排序：

1. 依赖和运行时升级
2. 配置外置化
3. 登录与密码安全
4. 替换 ``eval`` 转换逻辑
5. 测试、Lint 和发布纪律

当前约束
--------

当前仓库中可以看到以下约束：

- 仓库仍带有偏旧的打包假设
- ``pyproject.toml`` 内容过少，还没有承担现代打包元数据职责
- 文档和 CI 仍以 Python 3.9 为主
- ``zspider/confs/conf.py`` 中写死了本地和生产假设
- ``zspider/web.py`` 仍使用偏开发式的 Flask 启动方式
- 当前登录流程允许首个提交的账号直接成为管理员
- ``zspider/pipelines/publish.py`` 会执行来自配置的 Python ``eval``

这些问题决定了项目目前更适合在受控环境中维护，而不适合直接投放到“最新一切”的运行环境。

推荐改造阶段
------------

阶段 0：建立基线
~~~~~~~~~~~~~~~~

目标：

- 在不改行为的前提下，把当前系统稳定复现出来

交付项：

- 明确一套已知可用的基线环境
- 记录当前启动、测试和部署流程
- 准备一批代表性任务和样例数据，用于回归验证
- 明确下一步 Python 版本升级目标

阶段 1：工具链与运行时基础
~~~~~~~~~~~~~~~~~~~~~~~~~~

目标：

- 在不改产品行为的前提下，升级工程基础设施

主要工作：

- 统一本地开发 Python 版本
- 在 CI 中加入 lint、format、test 等质量门禁
- 建立依赖锁定策略
- 明确支持版本和淘汰窗口

应用化方向：

- 将仓库视为可部署应用，而不是待发布库
- 优先使用 ``python -m zspider.web`` 这类源码运行入口
- 仅在内部部署或构建需要时保留最小化打包能力

建议工具方向：

.. list-table::
   :header-rows: 1
   :widths: 24 76

   * - 关注点
     - 建议
   * - Packaging
     - ``pyproject.toml`` + setuptools 或 hatch
   * - Lint
     - ``ruff``
   * - Format
     - ``black``
   * - Tests
     - ``pytest``
   * - Type Check
     - ``mypy``，先覆盖关键边界
   * - 本地自动化
     - ``make`` 继续保留为兼容封装

阶段 2：配置与部署治理
~~~~~~~~~~~~~~~~~~~~~~

目标：

- 去掉代码中的环境假设，让服务启动方式明确且可迁移

主要工作：

- 把 MQ、缓存、数据库地址改成环境变量驱动
- 引入统一的 settings 层
- 区分开发、测试、生产环境
- 用 Docker Compose 明确定义本地依赖
- 用生产可用的 WSGI/ASGI 启动方式替换开发式 ``app.run``
- 增加健康检查和 readiness 文档

阶段 3：安全加固
~~~~~~~~~~~~~~~~

目标：

- 把项目的安全基线提升到现代内部平台的最低要求

主要工作：

- 将密码存储从直接 SHA-256 升级到 ``bcrypt`` 或 ``argon2``
- 移除“首次登录自动成为管理员”的逻辑
- 提供显式的管理员初始化命令或流程
- 检查 session、secret、CSRF 和管理员权限边界
- 为登录与鉴权补充回归测试

高优先级风险清理：

- 用受限表达式语言、白名单转换器或插件机制替换 ``eval``

阶段 4：API-first 边界重构
~~~~~~~~~~~~~~~~~~~~~~~~~~

目标：

- 让后台逻辑和模板渲染解耦

主要工作：

- 为任务、解析配置、结果查看和日志定义稳定 API
- 将业务逻辑从 request handler 提取到 service 层
- 降低模板与持久化模型的直接耦合
- 让当前 HTML 后台成为后端能力的一个客户端

阶段 5：运行时与数据平面优化
~~~~~~~~~~~~~~~~~~~~~~~~~~~~

目标：

- 在不改变产品模型的前提下提升可靠性和扩展性

主要工作：

- 明确 Dispatcher 与 Crawler 之间的消息 schema
- 给调度任务和抓取执行补齐状态模型
- 改进失败重试、错误队列或补偿机制
- 视实际收益考虑是否以 Redis 替代 Memcached
- 补充队列堆积、抓取成功率、解析失败率、发布失败率等指标

阶段 6：体验层现代化
~~~~~~~~~~~~~~~~~~~~

目标：

- 在技术核心稳定后再改进后台使用体验

可以考虑：

- 重构后台信息架构
- 提升任务创建和解析调试体验
- 增加任务激活前预览和校验
- 增加运行历史、差异视图和结果检查能力
- 改进调度状态和发布失败反馈

建议交付顺序
------------

一个小团队比较合理的推进顺序是：

1. 建立基线测试和可复现环境
2. 现代化打包、依赖和 CI
3. 配置外置化和部署稳定化
4. 加固认证并移除动态代码执行
5. 提取服务边界和内部 API
6. 优化运行可靠性和后台体验

非目标
------

以下事项不应作为第一阶段目标：

- 立即拆成微服务
- 仅为了“看起来新”而替换 Flask
- 全量重写 spiders 和 parsers
- 在没有明确收益前先做激进数据迁移
- 在 API 边界未稳定前先做重前端

成功标准
--------

当以下条件成立时，这轮改造才算成功：

- 新开发者可仅凭文档完成环境搭建
- CI 能稳定验证运行时和代码质量
- 生产配置全部外置
- 隐式管理员初始化和任意 ``eval`` 已移除
- 任务执行和后台操作具备足够的可观察性
- 项目不再依赖一套冻结的旧环境才能继续演进
