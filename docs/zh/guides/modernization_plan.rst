现代化改造计划（按定位自顶向下梳理）
====================================

本文不是单纯罗列技术 TODO，而是基于当前仓库实际状态，对 ZSpider 的后续改造做一次按“定位 -> 用户路径 -> 系统支撑 -> 工程治理”展开的重排。

这版计划有两个明确约束：

- 不改变项目作为“自托管抓取平台”的定位
- 略微提高用户侧体感优先级，但不以牺牲运行稳定性为代价


约定
----

- ``[x]`` 已完成
- ``[ ]`` 未完成
- ``[-]`` 当前阶段不做，后续再评估


一、定位先行：先回答 ZSpider 要继续成为什么
----------------------------------------

目标定位
~~~~~~~~

ZSpider 当前最合适的定位不是“可复用爬虫框架”，也不是“通用内容中台”，而是一套面向持续内容监控场景的自托管抓取应用：

- 有 Web 后台，可配置任务、字段、解析规则
- 有 Dispatcher，负责调度和任务投递
- 有 Crawler，负责消费队列并执行抓取
- 有结果存储、日志查看和发布链路

因此后续改造应围绕以下几条主线展开：

- 保留 ``web / dispatcher / crawler`` 三进程形态
- 保留任务、解析器、字段配置这套核心领域模型
- 优先降低“新增任务、调试任务、定位失败、恢复运行”的成本
- 在此基础上再逐步处理工具链、安全、边界和部署治理

成功标准
~~~~~~~~

如果这轮计划推进有效，最终应达到以下结果：

- 新用户能更快理解“怎么建任务、怎么测试、怎么启动、怎么看结果”
- 老用户遇到失败时，能更快定位是调度问题、抓取问题、解析问题还是发布问题
- 维护者能明确系统边界，不必在模板、路由、模型之间来回追踪逻辑
- 项目继续稳定运行在受控环境中，而不是为了“现代化”做大拆大改


二、基于仓库现状的复核结论
------------------------

已具备的基础
~~~~~~~~~~~~

从当前仓库看，底层基线已经比旧项目分析文档描述的更完整：

- [x] 项目定位已收口为“自托管应用”
- [x] 文档已重组为中英文结构
- [x] 本地入口已统一到 ``make`` / ``python -m`` / ``docker compose``
- [x] 已引入 ``settings`` 层，核心运行配置已可由环境变量驱动
- [x] 已区分 ``development / testing / production`` 环境
- [x] Web 进程已改为 WSGI 启动，而非 ``app.run``
- [x] 已提供 ``/healthz`` 和 ``/readyz`` 端点
- [x] 已提供显式管理员初始化命令 ``make bootstrap-admin``
- [x] CI 已包含 lint 和单元测试
- [x] 仓库已引入 ``ruff``
- [x] ``publish.py`` 中基于配置直接执行 ``eval`` 的高风险实现已被替换

当前仍然最值得优先处理的问题
~~~~~~~~~~~~~~~~~~~~~~~~~~~~

结合代码现状，真正还没收口的重点已经不是“项目能不能跑起来”，而是下面几类问题：

1. 用户主路径不够顺

   - 任务创建、字段配置、测试索引、测试文章、保存并启动，仍然耦在单个页面和表单行为里
   - 任务列表只能看到较粗粒度状态，离“快速判断任务现在为什么不工作”还有距离
   - 日志、运行状态、结果查看是分散入口，定位链路偏长

2. 运行状态可见性不足

   - Web 有健康检查，但 Dispatcher / Crawler 缺乏同等级的可观测入口
   - Dispatcher -> Crawler 的消息结构是隐式字典，不够稳定
   - 抓取、解析、发布各阶段缺少统一状态模型和运行历史

3. Web 层边界仍偏历史项目写法

   - ``handlers`` 中同时承担路由、表单装配、模型写入、任务重载等职责
   - 模板和模型绑定较深，不利于后续做更清晰的接口层
   - 错误反馈仍偏“页面闪消息 + 日志查看”，对用户不够直接

4. 工程治理尚未收尾

   - Python 版本策略还没有在文档与工具配置中彻底讲清楚
   - 仍在使用 ``unittest discover``，测试入口与文档存在不一致
   - ``pyproject.toml`` 目前只承载了少量工具配置
   - ``mypy``、更细的类型边界和部分部署文档仍未补齐


三、重排后的改造优先级
--------------------

这一版不再按“纯技术模块”拆 P0-P5，而是按业务落地顺序拆成四层。

L0：定位与主路径收口
~~~~~~~~~~~~~~~~~~~

这一层决定项目是不是更容易理解、也更容易持续维护，优先级最高。

- [x] 明确项目定位为“自托管抓取平台”
- [x] 保留三进程架构与配置驱动解析模型
- [ ] 在文档中明确一条标准用户主路径：

  - 创建任务
  - 配置解析规则与字段
  - 测试索引 / 测试文章
  - 保存并启动
  - 查看最近结果 / 最近日志
  - 判断失败落点并修复

- [ ] 统一“任务调试”与“任务运行”的心智模型，减少用户理解分叉
- [ ] 补一份“用户操作视角”的后台说明，而不只是开发 / 运维说明

L1：优先做用户可感知改进
~~~~~~~~~~~~~~~~~~~~~~~

这一层是本次计划里刻意略微前置的部分，但仍然只做高收益、低破坏的改动。

- [ ] 重构后台信息架构，优先梳理任务列表、任务编辑、结果查看、日志查看之间的跳转关系
- [ ] 改进任务创建流程，减少“一个大表单里塞满所有步骤”的理解成本
- [ ] 增强解析调试与预览能力，让“测试索引 / 测试文章”的结果更直接可读
- [ ] 增强任务状态展示，让用户能更快看出任务是否启用、是否最近运行过、最近一次大致结果如何
- [ ] 优化失败反馈，让常见失败至少能分成调度失败、抓取失败、解析失败、发布失败几类
- [ ] 优化日志与结果页面的联动，降低来回切页面排查的成本

四、建议的实施批次
----------------

如果按“先让产品更像一个可用的平台，再继续做深层治理”的思路推进，建议拆成以下三批。

第一批：优先解决用户能感知到的问题
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

目标：让用户更容易完成“配置 -> 测试 -> 启动 -> 观察 -> 排错”这条主路径。

- [x] 梳理后台页面信息架构，先明确任务列表、任务编辑、测试、日志、结果之间的主导航
- [x] 调整任务创建 / 编辑页交互，减少混杂按钮和隐式行为
- [x] 补一版更清晰的任务状态展示和失败分类
- [x] 增强测试结果展示，让解析调试更直接

第二批：补齐状态与边界
~~~~~~~~~~~~~~~~~~~~~~

目标：让系统本身更容易被观察，也让后续页面优化有稳定数据基础。

- [x] 定义任务运行历史模型
- [x] 定义 Dispatcher -> Crawler 消息 schema
- [x] 增加 Dispatcher / Crawler 健康与存活信号

第三批：收尾工具链、安全与部署治理

- [ ] 增加最小 ``mypy`` 覆盖

五、当前阶段的明确非目标
----------------------

- [-] 不做整仓重写
- [-] 不先拆成微服务
- [-] 当前阶段不做前后端大拆分
- [-] 不为了“现代化外观”先重做前端
- [-] 不在没有运行痛点前先做数据库迁移
- [-] 不把“发 PyPI”当目标


六、一句话结论
-------------

ZSpider 当前已经完成了“能以更现代的方式跑起来”的第一阶段，下一阶段更值得做的，不是继续堆底层名词，而是围绕“任务配置、调试、运行、排障”这条用户主路径，把状态可见性、失败反馈和 Web 边界一起收口；在这个基础上，再完成工具链、安全和部署治理的剩余工作。
