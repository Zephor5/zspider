页面探索实现记录
================

本文说明当前“页面探索辅助配置”的实际实现、任务创建主线，以及围绕真实站点验证后的结论。


适用范围
--------

当前记录覆盖以下内容：

- 入口页探索与文章页探索的实际职责
- 页面获取方式判断逻辑
- 页面预览、高亮与点选反馈
- 索引规则生成与文章字段生成的真实返回结构
- 针对 ``https://mil.news.sina.com.cn/`` 的验证结论


当前定位
--------

页面探索的定位不是让用户手工编写 XPath 或正则，而是让用户先看真实页面，再通过点选完成配置。

当前任务创建主线固定为：

1. 先确认入口页能找到真实文章链接
2. 再确认一篇文章能提取标题、正文、时间、来源
3. 最后检查测试结果并保存任务

页面探索负责完成下面三件事：

- 展示系统实际使用的页面预览
- 收集用户点选的正样本、反例和字段样本
- 把可执行规则写回任务表单


当前实现
--------

入口页探索
~~~~~~~~~~

当前入口页探索由 ``zspider/services/page_explorer.py`` 驱动，核心能力包括：

- 同时获取直接请求 HTML 和浏览器渲染 HTML
- 比较两种结果的正文相似度与链接重合度
- 给出“直接抓取”或“浏览器渲染”的建议
- 生成当前推荐获取方式下的页面预览
- 支持在预览中直接点击真实文章链接，自动加入正样本
- 支持把误命中的链接加入反例

入口页探索完成后，页面会更新：

- 建议抓取程序
- 任务名建议
- 入口页预览
- 右侧阶段状态


文章页探索
~~~~~~~~~~

当前文章页探索围绕以下字段工作：

- 标题
- 正文
- 时间
- 来源

文章页探索与入口页一样，按推荐获取方式展示预览，并支持：

- 点击真实节点
- 把当前节点标记为标题、正文、时间、来源
- 基于点选结果直接生成字段配置

入口规则确认后，页面会自动把最近一次确认过的正样本文章 URL 带入文章页输入框，减少重复输入。


获取方式判断
~~~~~~~~~~~~

当前获取方式判断分为两层：

1. 优先比较直接请求 HTML 与浏览器渲染 HTML
2. 如果浏览器结果不可用，再回退到脚本数、壳节点、正文长度、链接数量等静态启发式

当前判断结果会直接影响：

- 推荐的抓取程序
- 左侧页面预览来源
- 入口页规则生成所使用的 DOM
- 文章字段生成所使用的 DOM


页面预览与反馈
~~~~~~~~~~~~~~

当前预览支持三类反馈：

- 入口页初始高亮
- 按当前入口配置重新高亮
- 按当前文章字段配置重新高亮

当前预览中的点选结果会返回：

- 选中节点文本
- 节点 XPath
- 文本 XPath
- 正文容器 XPath
- 属性值 XPath

其中，用户只需要关注“点中了什么、要标成什么字段”，不需要自己消费这些底层数据。


索引规则生成
~~~~~~~~~~~~

当前索引规则生成入口为：

- ``POST /task/explore/index/generate``

对应服务函数为：

- ``generate_index_rule()``

当前主交互路径如下：

1. 在入口页预览中点击至少 2 个真实文章链接
2. 如果有误命中，把错误链接加入反例
3. 触发“生成索引规则”
4. 把生成结果写回右侧配置
5. 用测试索引验证真实抓取结果

当前实现中，索引规则已经拆成两类用途：

- 预览规则：只服务左侧高亮
- 解析规则：只服务 ``IndexParser`` 和真实抓取

这意味着：

- 左侧预览可以继续高亮 ``<a>`` 节点
- 右侧表单会写入可直接提取 URL 的 parser-ready 规则，例如 ``/@href``


文章字段生成
~~~~~~~~~~~~

当前文章字段生成入口为：

- ``POST /task/explore/article/generate``

对应服务函数为：

- ``generate_article_fields()``

当前交互方式是：

1. 在文章预览中分别点选标题、正文、时间、来源
2. 触发“生成文章字段”
3. 把可用字段写回右侧配置
4. 明确提示已写入字段和缺失字段

当前生成结果不再只按模型原始输出返回，而会追加真实落表校验：

- 哪些字段已经写入表单
- 哪些字段仍然缺失
- 当前结果对应的简要提示

如果 ``title``、``content``、``src_time``、``source`` 中仍有字段为空，页面不会再统一提示“四个字段都已写入配置”。


测试结果展示
~~~~~~~~~~~~

测试结果当前已经按“结果是否可用”来组织：

- 测试索引：展示真实 URL 和可选标题
- 测试新闻：逐项展示 ``url``、``title``、``content``、``src_time``、``source``
- 字段缺失时，直接说明当前没有解析出哪一项，并显示当前配置值

当前测试结果页的定位是：

- 先确认这份配置能不能用
- 再决定继续调整还是直接保存


实际验证结论
------------

验证站点
~~~~~~~~

当前重点验证过的真实站点：

- ``https://mil.news.sina.com.cn/``


站点结构特征
~~~~~~~~~~~~

该站点暴露出当前探索能力的典型难点：

- 索引页单个卡片中同时存在正文链接、评论链接、标签链接
- 列表区域中混有排行、推荐、广告等结构相似块
- 详情页存在多个标题类节点，例如 ``channel-logo``、``main-title``、``second-title``
- 时间、来源所在区域文本短、结构紧，容易误选外围容器


当前已验证通过的主路径
~~~~~~~~~~~~~~~~~~~~~~

围绕新浪军事，当前已经验证通过下面这条主路径：

- 探索入口页
- 点击真实文章链接加入正样本
- 生成入口规则并回填 ``/@href`` 形式的索引 XPath
- 探索一篇真实文章
- 点选标题、正文、时间、来源
- 生成文章字段并按真实结果补齐缺失项
- 执行“测试索引”“测试新闻”
- 保存任务并重新进入编辑页核对配置

这条路径已经可以在真实站点上跑通。


当前判断
--------

结合真实站点验证结果，当前结论如下：

- 页面探索已经可以承担任务创建主路径中的“看页面、点样本、回填配置”
- 复杂新闻站点最关键的不是“能不能生成一个 XPath”，而是“预览规则和解析规则是否一致”
- 文章字段生成最关键的不是模型是否返回了答案，而是答案是否真正写进了表单并能通过测试
- 对媒体运营者来说，测试结果页必须直接回答“这份配置现在能不能用”

当前更合适的定位是：

- 页面探索负责真实页面预览与样本采集
- 模型层负责在复杂结构中生成最终规则
- 测试结果负责验证规则是否真的可用
- 任务表单负责承接最终配置，而不是成为用户手工编写规则的主界面
