路线图：战略视图
================

本文从项目整体方向出发，回答一个更上游的问题：
在智能体、浏览器自动化、Tool Calling 已经越来越方便的今天，ZSpider 还要不要继续投入、应该往什么方向投入。

结论先写在前面：

**ZSpider 仍有持续演进的必要，但不适合把自己定义成“什么都能做的通用智能抓取平台”。**

更合适的方向是：

- 保留它在**长期稳定抓取、调度、结构化提取、运行治理**上的优势
- 弱化它在**通用交互、临时性探索、弱约束自动化**上的野心
- 优先补齐 internal API，把它逐步收敛成一个同时服务 **Agent 调用** 和 **用户使用** 的执行层与数据层组件


一、定位判断
------------

在 Agent 工具越来越强之后，很多过去需要专门写脚本或平台功能的场景，已经可以直接通过：

- 浏览器自动化
- Tool Calling
- LLM 推理 + 页面理解
- 聊天式任务编排

快速完成。

因此，ZSpider 如果沿着“通用采集 + 通用后台 + 通用自动化入口”去扩，很容易遇到两个问题：

1. 与 Agent 平台能力重叠

   - 临时抓一个网页
   - 看页面内容后再决定点哪里
   - 做一次性的人工协同操作
   - 根据上下文临场调整抓取逻辑

   这些事情，Agent 往往更灵活。

2. 平台复杂度持续上升

   - 后台要做更多交互能力
   - 状态机要覆盖更多不确定流程
   - 用户心智会从“配置型抓取平台”滑向“什么都想做的平台”

   这样会让系统边界越来越模糊，维护成本越来越高。

但这并不意味着 ZSpider 没价值了。恰恰相反，**在 Agent 不擅长长期稳定运行的地方，ZSpider 反而更有价值。**


二、战略边界
------------

这里的判断标准可以归纳成一句话：

**凡是 Agent 已经天然擅长的，ZSpider 就尽量不重复造轮子；凡是 Agent 难以长期稳定维护的，ZSpider 就重点做深。**

可以据此把能力分成三类。


三、战略主线
------------

1. 长周期、重复性、可审计的数据采集
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

这类事情最适合 ZSpider：

- 固定站点长期监控
- 定时抓取与增量抓取
- 结构化字段提取
- 多任务并发调度
- 失败重试、告警、运行历史留痕

原因很直接：

- 规则相对稳定
- 执行频率高
- 需要低成本长期运行
- 需要知道“什么时候开始坏了”
- 需要结果可追溯、可复盘

这部分是 Agent 常常能做出来、但不适合一直盯着做的工作。

对应重点包括：

- 强化任务调度、失败分类、重试与运行历史
- 强化任务结果的结构化展示与比对
- 强化任务级别的观测能力，而不是堆叠更多泛化交互能力


2. 面向垂直场景的“高可靠模板化采集”
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

ZSpider 不一定要追求“支持所有网站”，更值得做的是：

- 把已有高频站点做成高可靠模板
- 让字段、解析、发布链路更标准化
- 让用户能快速复用成熟采集方案

重点不在“功能看起来很多”，而在：

- 某类站点能稳定跑很久
- 某类内容抽取质量稳定
- 出问题时能快速定位具体阶段

对应重点包括：

- 优先沉淀几类代表性站点或任务模板
- 建立字段、解析器、结果校验的复用规范
- 把“模板可维护性”放到比“模板数量”更高的位置

为了降低配置门槛，用户侧更合适的交互不是直接书写 XPath / 正则，而是：

- 先输入入口页或文章页地址
- 由系统判断更合适的页面获取方式
- 在页面预览里确认候选链接、候选标题、候选时间或候选正文
- 再把确认结果写入规则配置

这类页面探索能力的定位是“辅助生成确定性规则”，而不是替代长期稳定的规则执行链路。


3. 作为执行底座，优先补齐 internal API
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

这是最值得认真考虑的方向。

未来更合理的分工不是让 ZSpider 自己变成完整智能体，而是：

- OpenClaw / Agent 负责理解需求、交互、临场决策
- ZSpider 负责稳定执行抓取任务、保存结果、提供运行状态

也就是把 ZSpider 往下面这些角色靠：

- 任务执行器
- 调度与队列层
- 结构化提取流水线
- 抓取结果存储与检索层
- 被 Agent 调用的内部服务

对应重点包括：

- 把 internal API 作为更高优先级的支撑面，而不是只把 Web 后台做厚
- 为任务创建、运行、查询状态、查询结果补齐稳定接口
- 明确“哪些动作适合后台手工做，哪些动作适合 API 调用”
- 逐步把 Web 后台从“唯一入口”收敛为“管理入口之一”

这条路线分成两条并行但共用底层能力的子路线：

1. 面向 Agent 的调用路线

   - 重点是稳定的任务创建、触发执行、查询状态、获取结果接口
   - 重点是让外部 Agent 可以把 ZSpider 当作高可靠执行器，而不是重复实现调度、存储和运行治理

2. 面向用户的使用路线

   - 重点是让后台、CLI 或后续其他入口都建立在同一套 internal API 之上
   - 重点不是让用户直接面对更多技术概念，而是让“配置、测试、启动、排障”的体验建立在更稳定的边界上

这两条路线不是二选一，而是同一套 internal API 的两个消费面：

- 面向 Agent：强调可调用、可编排、可集成
- 面向用户：强调一致性、可维护性、可演进


4. 可观测性与运维体验
~~~~~~~~~~~~~~~~~~~~~~

如果 ZSpider 的定位是自托管抓取平台，那运维体验就是产品能力的一部分。

相比继续加花哨能力，更值得做的是：

- 看得出任务有没有在跑
- 看得出失败发生在哪一层
- 看得出最近结果有没有异常
- 看得出队列有没有堆积
- 看得出某个站点是不是整体失效了

对应重点包括：

- 强化任务状态、运行历史、错误分类、健康检查
- 增加围绕“任务维度”的日志与指标联动
- 优先降低排障成本，而不是优先扩展配置项数量


四、降级方向
------------

1. 通用型 Agent 交互能力
~~~~~~~~~~~~~~~~~~~~~~~~

例如：

- 在后台里做大量临场式交互流程
- 试图把各种探索式网页操作都塞进统一配置模型
- 追求“像人一样看页面再决定下一步”的执行体验

这类能力并不是做不到，而是性价比不高。

更合理的方式通常是：

- 让 Agent 负责探索和判断
- 让 ZSpider 接手确定性的、可重复的执行链路

但这并不排斥在 ZSpider 后台中提供有限的页面探索辅助能力。合理边界是：

- 可以做入口页、文章页的候选识别与页面预览
- 可以做页面获取方式的自动判断与建议
- 可以做候选规则回填
- 不把后台扩展成覆盖任意网页交互的通用浏览器自动化平台


2. 过度追求“大而全”的平台叙事
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

需要避免把项目描述成：

- 通用内容平台
- 通用自动化平台
- 通用智能采集平台
- 一站式所有问题解决方案

这种叙事短期看起来很大，长期会让边界失控。

更合适的外部表达是：

**面向持续监控与结构化采集场景的自托管抓取平台。**

必要时再补一句：

**可作为 Agent 系统的稳定执行层。**


3. 为了“现代感”而做的大规模表层重写
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

例如：

- 还没解决核心主路径之前，就先全面重做前端
- 还没理顺内部边界之前，就先拆复杂前后端架构
- 还没证明收益之前，就大规模替换已有运行链路

这些改动很容易消耗大量时间，却不能直接提升项目的独特价值。

对应重点包括：

- 界面只做必要的可用性改进
- 架构只做支持长期维护的边界收口
- 把主要投入放在稳定性、状态可见性、接口化能力上


五、投入优先级
--------------

下面给出更明确的方向性清单。

A. 应继续投入
~~~~~~~~~~~~~

- [ ] 完善任务调度、重试、失败分类、运行历史
- [ ] 完善任务结果、日志、状态的联动视图
- [ ] 沉淀适合长期运行的高价值采集模板
- [ ] 优先补齐 internal API，让 ZSpider 同时可被外部系统稳定调用，也可作为用户入口的一致底层
- [ ] 提升部署、监控、健康检查、排障体验

B. 应谨慎投入
~~~~~~~~~~~~~

- [ ] 后台交互体验优化
- [ ] 任务创建流程优化
- [ ] 更清晰的解析调试、页面探索与候选规则预览能力
- [ ] 适度的界面现代化

这些事情可以做，但应服务于“更容易配置、更容易排障”，而不是为了追求表面产品感。

C. 应停止扩张或降级优先级
~~~~~~~~~~~~~~~~~~~~~~~~~

- [ ] 停止把 ZSpider 往通用智能体平台方向叙述
- [ ] 停止尝试覆盖大量临时性、探索式、弱确定性网页操作
- [ ] 停止为“大而全”愿景堆叠抽象层
- [ ] 降低那些 Agent 已经能低成本完成的能力的产品优先级


六、定位表述
------------

项目定位可以表述为：

**ZSpider 是一套面向持续监控、结构化提取与长期运行治理场景的自托管抓取平台。**

如果需要体现与 Agent 的关系，可以补充为：

**它既可以作为独立运行的抓取系统，也可以作为 OpenClaw 等 Agent 系统的稳定执行层与数据层组件；而 internal API 是连接这两条路线的关键边界。**


七、一句话总结
--------------

ZSpider 不是没有继续投入的必要，而是不适合按“通用智能采集平台”的方向投入。

更值得走的路线是：

**收敛定位，优先补齐 internal API，强化稳定执行、长期运行、结构化提取和可观测性，让 ZSpider 同时服务 Agent 调用与用户使用。**
