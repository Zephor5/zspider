路线图：执行视图
================

本文按“定位 -> 用户路径 -> 系统支撑 -> 工程治理”的顺序展开，描述 ZSpider 当前执行层的改造重点。

这份执行视图有两个基本约束：

- 不改变项目作为“自托管抓取平台”的定位
- 略微提高用户侧体感优先级，但不以牺牲运行稳定性为代价

- ``[x]`` 已完成
- ``[~]`` 部分完成，已落地一部分，但还没有完全收口

一、定位与目标
--------------

目标定位
~~~~~~~~

ZSpider 当前最合适的定位不是“可复用爬虫框架”，也不是“通用内容中台”，而是一套面向持续内容监控场景的自托管抓取应用：

- 有 Web 后台，可配置任务、字段、解析规则
- 有 Dispatcher，负责调度和任务投递
- 有 Crawler，负责消费队列并执行抓取
- 有结果存储、日志查看和发布链路

执行重点围绕以下几条主线展开：

- 保留 ``web / dispatcher / crawler`` 三进程形态
- 保留任务、解析器、字段配置这套核心领域模型
- 优先降低“新增任务、调试任务、定位失败、恢复运行”的成本
- 在此基础上再逐步处理工具链、安全、边界和部署治理

成功标准
~~~~~~~~

如果这轮计划推进有效，最终应达到以下结果：

- 新用户能更快理解“怎么建任务、怎么测试、怎么启动、怎么看结果”
- 老用户遇到失败时，能更快定位是调度问题、抓取问题、解析问题还是发布问题
- 维护者能明确系统边界，不必在模板、路由、模型之间来回追踪逻辑
- 项目稳定运行在受控环境中，而不是为了“现代化”做大拆大改

当前仍然最值得优先处理的问题
~~~~~~~~~~~~~~~~~~~~~~~~~~~~

结合代码现状，当前重点不在“项目能不能跑起来”，而在下面几类问题：

1. 用户主路径不够顺

   - 任务创建、字段配置、测试索引、测试文章、保存并启动，仍然耦在单个页面和表单行为里
   - 任务列表只能看到较粗粒度状态，离“快速判断任务现在为什么不工作”还有距离
   - 日志、运行状态、结果查看是分散入口，定位链路偏长

2. 运行状态可见性不足

   - Web、Dispatcher、Crawler 虽然都已有健康或存活信号，但任务维度的状态联动仍然不足
   - Dispatcher -> Crawler 虽然已有显式消息 schema，但围绕消息的观测、重试和补偿还没补齐
   - 抓取、解析、发布虽已有统一状态码和基础运行历史，但离用户可直接消费的运行视图还有距离

3. Web 层边界仍偏历史项目写法

   - ``handlers`` 中同时承担路由、表单装配、模型写入、任务重载等职责
   - 模板和模型绑定较深，不利于后续做更清晰的接口层
   - 错误反馈仍偏“页面闪消息 + 日志查看”，对用户不够直接

4. 工程治理尚未收尾

   - Python 版本策略还没有在文档与工具配置中彻底讲清楚
   - 仍在使用 ``unittest discover``，测试入口与文档存在不一致
   - ``pyproject.toml`` 目前只承载了少量工具配置
   - ``mypy``、更细的类型边界和部分部署文档仍未补齐


三、一次真实试用记录
--------------------

为避免路线图只停留在代码阅读层，这里记录一次按真实用户路径完成的后台试用。

试用路径
~~~~~~~~

本次试用实际走了下面这条路径：

- 使用管理员账号登录后台
- 打开任务列表、添加任务、运行历史、数据记录页面
- 在“添加任务”页按真实站点填写任务
- 先执行“测试索引”，再执行“测试新闻”

试用站点
~~~~~~~~

本次选用的真实站点是：

- ``https://blog.python.org/``

原因是它符合 ZSpider 当前最擅长的目标形态：

- 首页可稳定发现文章链接
- 详情页结构清晰
- 标题、正文、时间、来源都能用配置方式抽取

试用时采用的任务配置
~~~~~~~~~~~~~~~~~~~~

本次在后台实际填写并验证通过的一组配置如下：

- Spider：``browser``
- Parser：``IndexParser``
- 入口 URL：``https://mil.news.sina.com.cn/``
- 索引 XPath：``//div[contains(@class,'ty-cardlist-w')]//h3[@class='ty-card-tt']/a[@node-type='sima-click-0']/@href``
- 标题 XPath：``//h1[@class="main-title"]//text()``
- 正文 XPath：``//*[@id="article"]//p//text()``
- 时间 XPath：``//div[@class="date-source"]//span[@class="date"]//text()``
- 来源 XPath：``//div[@class="date-source"]//a[@class="source"]//text()``

试用结果
~~~~~~~~

- “测试索引”成功返回多条真实文章链接
- “测试新闻”成功返回 1 条真实文章字段预览
- 预览结果中已能看到 ``url``、``title``、``content``、``src_time``、``source``

这说明当前项目的“复杂新闻站点配置型抓取”主路径已经可以在真实站点上跑通。

试用中暴露出的真实问题
~~~~~~~~~~~~~~~~~~~~~~

这次试用过程中实际发现并确认了两类会直接影响用户体验的问题：

1. 入口规则的预览语义和解析语义不一致

   - 入口页高亮可以直接使用 ``<a>`` 节点 XPath
   - ``IndexParser`` 实际需要的是能直接提取 URL 的规则，例如 ``/@href``
   - 用户看到高亮正确，但测试时可能拿到拼接后的 HTML 片段，而不是真实 URL

2. 文章字段生成提示和真实落表结果可能不一致

   - 页面提示“四个字段都已写入配置”
   - 但 ``src_time``、``source`` 这类短文本字段仍可能为空或指向错误容器
   - 用户只有在“测试新闻”时才会发现结果不完整

围绕这两类问题，当前优化重点已经转向：

- 把预览规则和解析规则彻底分开
- 在字段生成后按真实表单值做校验
- 让测试结果直接表达“哪些字段可用、哪些字段缺失”

从试用得到的结论
~~~~~~~~~~~~~~~~

这次真实试用进一步验证了路线图前面的判断：

- 当前最影响体感的不是“系统能不能跑”，而是“用户能不能顺着主路径完成配置和验证”
- “添加任务”页虽然已经开始收口，但预览、回填、测试、保存之间仍有语义断层
- 一旦把规则语义和测试结果统一起来，ZSpider 在“复杂新闻站点 -> 配置化抽取”场景下是能直接工作的

因此，接下来的高优先级仍然应该放在：

- 改进任务创建流程
- 统一“任务调试”与“任务运行”的心智模型
- 把测试结果、保存动作、运行历史、日志结果进一步串成闭环

接下来的任务创建主路径，不应再继续强化“用户先写 XPath / 正则，再测试”的模式，而应围绕下面这条顺序组织：

- 用户先输入入口页地址
- 系统先判断页面更适合直接抓取还是浏览器渲染
- 页面探索面板展示入口页预览和候选链接区域
- 用户先在页面预览中确认候选，再把结果写入任务配置
- 文章页字段配置沿用同一套“预览 -> 候选 -> 回填”的交互方式

围绕这条主线，当前后台已经进一步落实了下面这组改动：

- 任务创建 / 编辑页已按“基础信息 -> 解析配置 -> 字段规则 -> 测试并保存”重组页面结构
- 测试结果页已补充“返回继续编辑”“继续测试下一步”“直接保存”“保存并启动”动作
- 测试结果现在会保留整份表单内容，测试和保存已经形成连续的用户路径
- 任务创建页右侧已补充步骤提示与测试 / 保存关系说明，减少“当前处于哪一步”的理解成本

当前这组改动已经把测试和保存串起来，但还需要继续把“页面探索辅助配置”真正变成稳定主路径，而不是一组看起来可用的辅助面板。

  仍欠缺：

  仍欠缺：

  - 失败分类还没有贯穿任务列表、详情、日志、结果这几个核心页面

  - 页面探索面板已能展示候选链接区域、建议 XPath 和页面预览
  - 用户已可把候选链接 XPath 直接回填到右侧配置表单
  - 用户已可从入口页候选直接带入一篇样例文章，继续探索标题、正文、时间和来源候选
