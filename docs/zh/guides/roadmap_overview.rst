路线图：总览
============

如果把 ZSpider 的演进路线看成一张自顶向下展开的导图，那么这份总览回答的是：

- 项目要成为什么
- 为什么继续投入
- 优先往哪些方向投入
- 这些方向最终要落到哪些能力建设与执行事项上

这套路线图由总览与两个视图共同构成：

- ``roadmap_strategy``：更偏上层，回答“方向是否正确、边界是否清晰、投入是否值得”
- ``roadmap_execution``：更偏下层，回答“能力缺口是什么、当前做到哪一步、下一步怎么推进”


一、定位与边界
--------------

定位
~~~~

ZSpider 当前更适合被定义为：

**一套面向持续内容监控、结构化提取与长期运行治理场景的自托管抓取平台。**

它的核心形态仍然是：

- Web 后台
- Dispatcher 调度
- Crawler 执行
- 结果存储与发布链路

它的核心价值不在“临场探索”或“通用自动化”，而在：

- 长期稳定执行
- 可配置的结构化提取
- 多任务调度与运行治理
- 可追溯、可排障、可维护

它的用户侧入口应围绕“页面探索辅助配置”展开，而不是让用户先面对内部实现概念：

- 先输入入口页地址
- 由系统判断更合适的页面获取方式
- 在页面预览中确认候选链接或候选字段
- 再把确认结果写入任务配置

非目标
~~~~~~

这套路线图明确排除下面这些方向：

- 通用智能体平台
- 覆盖一切探索式网页交互的通用自动化平台
- 为了“现代感”而做的大规模表层重写
- 在没有明确收益前做整仓重构或微服务化


二、目标与成功标准
------------------

用户目标
~~~~~~~~

- 新用户更快理解如何创建、测试、启动和排查任务
- 老用户更快判断问题发生在调度、抓取、解析还是发布阶段
- 使用者不需要跨很多页面和隐式规则才能完成主路径

系统目标
~~~~~~~~

- 保持三进程架构与配置驱动解析的核心模型稳定
- 强化任务维度的状态可见性、失败反馈和运行历史
- 让任务执行链路更容易被观察、复盘和调用

工程目标
~~~~~~~~

- 逐步理清 Web、service、internal API 的职责边界
- 补齐工具链、安全与部署治理
- 控制改造节奏，避免“为了现代化而现代化”


三、战略主线
------------

1. 长周期、重复性、可审计的数据采集
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

强化 ZSpider 在长期稳定抓取上的优势，而不是和 Agent 在临时性探索场景正面重叠。

2. internal API 优先
~~~~~~~~~~~~~~~~~~~

把 internal API 作为下一阶段最关键的支撑面之一，让同一套底层能力同时服务：

- 面向 Agent 的调用路线
- 面向用户的使用路线

3. 可观测性与运行治理
~~~~~~~~~~~~~~~~~~~~

围绕任务维度强化：

- 运行状态
- 失败分类
- 运行历史
- 日志与结果联动

4. 高可靠模板化采集
~~~~~~~~~~~~~~~~~~

不追求覆盖所有站点，而是优先沉淀少数高价值、高复用、高稳定性的任务模板。


四、能力地图
------------

这一层把战略主线拆成能力建设面。

1. 用户主路径能力
~~~~~~~~~~~~~~~~

围绕“配置 -> 测试 -> 启动 -> 观察 -> 排错”主路径，补齐：

- 后台信息架构
- 任务创建与编辑体验
- 入口页与文章页的页面探索能力
- 页面获取方式自动判断与候选规则生成
- 测试与预览能力
- 失败反馈与排障体验

2. 任务运行能力
~~~~~~~~~~~~~~

围绕任务执行闭环，补齐：

- 任务运行历史
- 阶段状态模型
- 错误码与错误分类
- 重试、补偿与异常恢复

3. internal API 能力
~~~~~~~~~~~~~~~~~~~~

围绕统一边界，补齐：

- 任务创建 / 更新 / 触发接口
- 状态查询接口
- 结果查询接口
- 日志与运行记录查询接口

4. 平台治理能力
~~~~~~~~~~~~~~

围绕长期维护，补齐：

- Python 版本策略
- 测试工具链
- 类型边界
- 部署与健康检查
- 安全配置与权限边界

六、一句话总结
--------------

从上往下看，这份路线图的逻辑是：

**先明确 ZSpider 是什么，再明确为什么继续做、重点往哪里做，最后把这些方向落实到 internal API、运行治理、用户主路径和工程治理等具体能力建设上。**
